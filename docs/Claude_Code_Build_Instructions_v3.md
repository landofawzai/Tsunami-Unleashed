## System-Wide Integration Standard

This section defines how all six automation pillars (Content Creation, Content Repurposing, Distribution, Communication, Administration, Discipling/Coaching) and the Translation Pipeline interoperate. Each pillar is a standalone system â€” separate codebase, separate deployment, no shared dependencies. They communicate through Google Drive (shared filesystem) and Pabbly Connect (event routing).

### The Integration Contract

Every pillar follows three rules:

1. **Read from and write to Google Drive using agreed folder paths**
2. **Include a metadata sidecar JSON file with every content piece**
3. **Pabbly Connect routes events between pillars â€” pillars never call each other directly**

This means any pillar can be rebuilt, replaced, or taken offline without breaking the others. If the Repurposing pillar goes down, the Distribution pillar still picks up anything manually placed in the Derivatives folder. If the Translation pillar is replaced with a completely different tool, as long as it drops files into `/Translations/{lang}/` with a valid sidecar, everything downstream works.

### Metadata Sidecar Format

Every content file in Google Drive is accompanied by a `.meta.json` file with the same base name. Example: `vine-and-branches.mp4` is paired with `vine-and-branches.meta.json`.

```json
{
  "contentId": "unique-id-generated-by-creating-pillar",
  "title": "The Vine and the Branches - John 15 Teaching",
  "contentType": "sermon",
  "language": "en",
  "sourceFile": "vine-and-branches.mp4",
  "mimeType": "video/mp4",
  "createdBy": "content_creation",
  "createdAt": "2026-02-09T14:30:00Z",
  "parentContentId": null,
  "isDerivative": false,
  "derivativeType": null,
  "tags": ["john-15", "discipleship", "abiding"],
  "targetLanguages": ["hi", "bn", "ur", "ta", "te"],
  "flags": {
    "readyForRepurposing": true,
    "readyForTranslation": true,
    "readyForDistribution": false,
    "highValueForSyndication": true,
    "needsHumanReview": false
  },
  "processingHistory": [
    {
      "pillar": "content_creation",
      "action": "created",
      "timestamp": "2026-02-09T14:30:00Z"
    }
  ]
}
```

**Field rules:**
- `contentId`: Generated by whichever pillar creates the item. Must be globally unique (use cuid or uuid).
- `createdBy`: One of `"content_creation"`, `"repurposing"`, `"translation"`, `"distribution"`, `"communication"`, `"discipling"`, `"manual"`.
- `parentContentId`: Links derivatives and translations back to their source. The Distribution pillar uses this to group related content.
- `flags`: Each pillar reads the flags relevant to it and updates the ones it owns. The creating pillar sets initial flags. Downstream pillars update them after processing.
- `processingHistory`: Append-only log. Each pillar adds an entry when it processes the item. This creates an audit trail without pillars needing to communicate directly.
- `highValueForSyndication`: When true, the Distribution pillar routes this to Tier 4 native upload queue (Ubot batch-uploads to satellite accounts). Set by Content Creation or Repurposing for the most impactful content.

Pabbly is the central nervous system. It watches Google Drive folders and routes events between pillars. No pillar triggers another pillar directly.

**Pabbly workflow naming convention:**
- `ROUTE-[source]-to-[destination]` (e.g., `ROUTE-SourceContent-to-Repurposing`, `ROUTE-Derivatives-to-Distribution`)

- `MAINTENANCE-[schedule]` (e.g., `MAINTENANCE-DailyMetrics`)

### Cross-Pillar Content ID Tracking

When a pillar processes content, it preserves the `contentId` from the metadata sidecar. This means:
- A sermon created by Content Creation (`contentId: "abc123"`) keeps that ID through Repurposing, Translation, and Distribution
- Derivatives reference it via `parentContentId: "abc123"`
- Translations reference it via `parentContentId: "abc123"` with their own unique `contentId`
- The Distribution dashboard can trace any piece of content back to its origin
- The Communication pillar can reference content IDs when responding to inquiries

No central database coordinates this. The `contentId` in the sidecar IS the coordination mechanism.

### How Sidecars Map to Dashboard Webhooks

The metadata sidecar (`.meta.json`) is the universal format across all pillars. The Distribution dashboard webhooks (Session 2) accept a simpler flat payload. This is by design â€” Pabbly Connect reads the sidecar and maps relevant fields to the webhook call. Not all sidecar fields are sent to the dashboard. The dashboard tracks distribution status, not full content metadata.

Example mapping (Pabbly reads sidecar â†’ calls webhook):
- Sidecar `contentId` â†’ not sent at initial intake (dashboard generates its own `id`), but used as `parentContentId` for derivatives/translations
- Sidecar `title` â†’ webhook `title`
- Sidecar `contentType` â†’ webhook `contentType`
- Sidecar `language` â†’ webhook `language`
- Sidecar `flags.readyForDistribution` â†’ Pabbly uses this to decide whether to trigger Distribution workflows (never sent to dashboard)
- Sidecar `flags.highValueForSyndication` â†’ Pabbly uses this to decide whether to create a Syndication Upload Queue JSON (never sent to dashboard)
- Sidecar `processingHistory` â†’ not sent to dashboard; the dashboard maintains its own timeline via DistributionLog entries

The sidecar is the source of truth for cross-pillar coordination. The dashboard webhook payloads are the source of truth for distribution tracking. They overlap but serve different purposes.

## Architecture Context: Four-Tier Distribution Strategy

This system uses a four-tier distribution architecture designed for resilience and persecution resistance:

**Tier 1 â€” RSSground Private Feeds (150 lifetime slots)**
The vault. Depth/discipleship content only. Core sermons, foundational teachings. Protected by RSSground's infrastructure. These are the most valuable feeds and hardest to replace. Integrated natively with Pabbly Connect. Never waste a Tier 1 slot on derivative or ephemeral content.

**Tier 2 â€” External/Self-Built Feeds (unlimited, low cost)**
The expansion layer. Derivatives, additional languages, topical series, partner content. Hosted on Ghost CMS, Cloudflare Workers, or custom-built RSS generators. Cheap to scale, easy to replace if a host goes down.

**Tier 3 â€” Platform-Native Distribution (most vulnerable)**
The discovery layer. Branded social media, messaging apps, video platforms. If any platform disappears, Tiers 1 and 2 continue delivering. Followr handles most social platforms via Ubot Studio automation; Pabbly handles messaging and edge cases; Robomotion covers platforms Followr doesn't support.

**Tier 4 â€” IFTTT Syndication Networks (fingerprint-free amplification)**
The scattering layer. Hundreds of unbranded satellite accounts (Blogger, WordPress, Tumblr, etc.) organized into multi-ring IFTTT syndication networks. Two content streams: (1) IFTTT autopilot pulls from Tier 1/2 RSS feeds and auto-posts text + images with zero intervention, (2) Ubot Studio batch-uploads native video/audio to satellite accounts using the master login spreadsheet. All accounts use unique personas â€” different names, bios, locations, profile images. No backlinks, no attribution, no fingerprints tracing back to HQ. Content appears to originate from independent local believers. CC0 license makes this legally clean. One-time setup cost per language (~$50-150 for a 2-tier network of 75-200+ accounts). After setup, runs forever with near-zero maintenance.

**Resilience logic:** Tier 3 ban â†’ Tiers 1+2+4 unaffected. Tier 4 accounts get suspended â†’ other rings continue, rebuild cheaply. Tier 2 host down â†’ spin up another, Tier 1 safe. RSSground disappears â†’ Tier 2 feeds continue, rebuild Tier 1 on new infrastructure. HQ disappears â†’ Tier 4 satellite accounts keep auto-posting from RSS feeds indefinitely. No single point of failure.

## Tool Responsibility Matrix

This system uses eight tools, each with a specific non-overlapping role:

| Tool | Role | Tier |
|---|---|---|
| **RSSground** | Tier 1 RSS feeds only. Depth content. No social posting. | Tier 1 |
| **Ghost/Cloudflare** | Tier 2 external feeds. Expansion content. | Tier 2 |
| **Followr** | Social media scheduling & posting UI. Handles Facebook, Instagram, X, LinkedIn, Pinterest, TikTok, YouTube, Medium. | Tier 3 |
| **Ubot Studio** | Browser automation bot. Three jobs: (1) Logs into Followr and schedules Tier 3 branded posts. (2) Batch-uploads native video/audio to Tier 4 satellite accounts. (3) Calls dashboard webhooks after posting. | Automation glue (Tier 3 + Tier 4) |
| **Robomotion** | Browser automation for edge-case platforms Followr doesn't cover: Rumble, Odysee, Signal. Backup automation layer. | Automation glue (Tier 3) |
| **Pabbly Connect** | Orchestration. Google Drive triggers â†’ routes to RSSground or external feeds â†’ drops post data into Followr Queue â†’ calls dashboard webhooks. | Orchestration |
| **IFTTT** | Tier 4 syndication autopilot. Watches Tier 1/2 RSS feeds â†’ auto-posts text + images to hundreds of unbranded satellite accounts. Runs independently after one-time setup. No dashboard integration needed. | Tier 4 |
| **Dashboard** | Tracks all content across all tiers. Receives webhooks from Pabbly, RSSground, Ubot, and Robomotion. Tier 4 operates independently â€” no dashboard tracking (by design, for fingerprint isolation). | Monitoring |

**The Followr Queue (Tier 3):** Pabbly drops social media post data (caption, media URL, platform targets, tags) into a Google Drive folder called "Followr Queue". Ubot Studio monitors this folder, picks up new items, logs into Followr, schedules the posts, then calls the dashboard webhook to report success/failure.

**The Syndication Upload Queue (Tier 4):** For high-value content (teaching videos, worship audio, powerful testimonies), Pabbly drops native upload instructions into a "Syndication Upload Queue" folder. Ubot Studio picks these up and batch-uploads video/audio files natively to satellite accounts using the master login spreadsheet. Unlike the IFTTT autopilot (which handles text + images automatically), this gives satellite accounts rich native media that looks completely organic. Runs 2-5 times per week on the highest-impact content.

**Why Ubot instead of Followr API:** Followr's API costs $99/month. Ubot Studio (already owned, unlimited license) can operate Followr's web UI like a human â€” uploading content, selecting platforms, applying Autopilot tags, scheduling posts. The tradeoff is that UI changes in Followr may occasionally require bot maintenance (~1-2 hours per quarter).

**Why Tier 4 is NOT in the dashboard:** Fingerprint isolation. The dashboard tracks Tiers 1-3 (your branded infrastructure). Tier 4 satellite accounts are deliberately disconnected â€” no webhooks, no tracking, no digital trail linking them to HQ. If someone accesses your dashboard, they see zero evidence of the syndication network. IFTTT handles the autopilot. Ubot handles the native uploads. The master login spreadsheet lives in an encrypted, offline location.
