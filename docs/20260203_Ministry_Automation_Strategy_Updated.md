# 5. Automation Architecture

## Core Design Principle

Every automated touchpoint is engineered with ONE question: "What will make this person want the local, embodied, Jesus-centered relationship that automation can't provide?"

The system inverts typical approaches:
- Conventional: Build local → add online as afterthought
- Lazy digital: Build online audience → hope they show up
- This system: Automation creates omnipresence AND engineered tension toward local resolution

Success metric: People leaving content streams for local relationships.
The goal is app ABANDONMENT, not engagement.

## Integration Architecture

The six automation pillars below are each built as standalone systems — separate codebases, separate deployments, no shared dependencies. They communicate through Google Drive (shared filesystem) and Pabbly Connect (event routing). A metadata sidecar file travels with every content piece, carrying its unique ID, processing flags, and audit trail. No pillar calls another directly. Any pillar can be rebuilt or replaced without breaking the others. This mirrors the biblical framework: decentralized, no single point of failure, no control dependency. Full technical details in Section 12.

# 6. Distribution Strategy: RSS + Platform Implementation

*Implementation note: This section describes the distribution principles and platform strategy. The specific four-tier technical architecture (RSSground, external feeds, branded platforms, IFTTT syndication networks) and the tool stack (Followr, Ubot Studio, Robomotion, Pabbly Connect, IFTTT) are detailed in Section 12: Cross-Pillar Integration Architecture and in the Distribution System Dashboard build instructions.*

## Why RSS Is Strategic Genius

RSS (Really Simple Syndication) is the most underrated distribution technology and perfectly aligned with the biblical framework.

Strategic Advantages:
- Completely decentralized (no central control point)
- Impossible to censor (it's just a file)
- Free forever (open standard)
- Platform-independent (works everywhere)
- Algorithmically neutral (subscribers get everything)
- Proven at billion-person scale (entire podcasting industry runs on RSS)

Game Theory Result:
- Governments can ban YouTube → content still accessible via RSS
- Platforms can change algorithms → RSS has no algorithm
- Can't "shadow ban" → direct delivery to subscribers
- Organization shutdown doesn't stop content distribution

This is the most persecution-resistant distribution technology that exists.

## RSS Implementation

Architecture:
Content Creation → RSS Feed (master) → Auto-distribute to:
├─ Podcast apps (Spotify, Apple, Google, etc.)
├─ RSS readers (Feedly, Inoreader)
├─ Email subscriptions (daily/weekly digests)
└─ Social media (automated posts from RSS)

Feed Structure:
- Separate feeds by content type (sermons, teachings, articles)
- Separate feeds by language (English, Hindi, Bengali, etc.)
- Result: ~200 total feeds (4 content types × 50 languages)
- Cost to host: ~$20/month

Distribution Process:
1. Create content once in English
2. Add to RSS feed (10 minutes)
3. AI translates to 50 languages automatically
4. Publishes to 50 language-specific RSS feeds
5. Subscribers in all languages receive automatically
6. Platforms pull from RSS and auto-post
7. Total ongoing HQ time: 10 minutes per content piece

Tools:
- Platform: Anchor.fm (free, auto-distributes) or Ghost CMS ($9/month)
- Translation: Claude/GPT-4 API ($300/month for aggressive volume)
- Automation: GitHub Actions (free) or Zapier ($30/month)

Field Leader Usage:
- Subscribe to RSS feed
- Download episodes offline
- Share via Bluetooth/local network (no internet needed)
- Can create their own RSS feed with adapted content (CC0 enables this)

# 7. Translation Pipeline: AI-Assisted, Field-Executed

## Strategic Approach

We're NOT building HQ translation team. We're enabling field leaders to translate themselves.

Why This Works:
- CC0 license = anyone can translate without permission
- AI provides quality first draft in seconds
- Local speakers refine for cultural context
- Result: Translation happens at point of need, not centrally
- Scales infinitely without adding HQ staff

Biblical Alignment:
- Indigenous adaptation (contextualization explosion)
- Local ownership (not waiting for HQ approval)
- Decentralized by design (no bottleneck)
- Reproduction enabled (others can translate the translations)

## Three-Pass Translation Workflow

Pass 1: AI Translation (30 seconds)
- Input: English content
- Tool: Claude API or GPT-4
- Output: Draft in target language
- Cost: ~$5 per sermon across 50 languages

Pass 2: Local Speaker Review (10-30 minutes)
- Local speaker refines AI draft
- Corrects cultural context
- Adds local idioms
- Ensures natural language flow
- This is where contextualization happens

Pass 3: Elder/Pastor Theological Review (Optional, 15-30 minutes)
- Checks biblical accuracy
- Ensures reproducibility
- Validates simplicity
- Only for core doctrinal content

Total Time: AI + local refinement = hours instead of weeks HQ Involvement: Zero (after tool is built)

## Translation Enablement Tool

Simple web interface that anyone can use:

Features:
- Paste English content → Select target language → Get AI draft
- Side-by-side editing (English + target language visible)
- Theological glossary built in (key terms translated consistently)
- Previous translations as reference
- Export to ready-to-use formats (docx, video subtitles, audio scripts)

Access:
- No login required (anonymous access)
- No approval needed
- No certification gatekeeping
- Results immediately CC0

Open Source:
- Tool code on GitHub
- Others can run their own instances
- Improvements contributed back
- Truly decentralized translation

Cost to Build: $3-5K one-time
Monthly API Cost: $300 for aggressive translation volume HQ Maintenance: 2-5 hours/month

## Quality Control Without Control

How do we maintain accuracy without centralized approval?

Method 1: Simplicity as Quality Control
- Simple, clear content translates reliably
- Complex theology = translation errors
- Reproducibility requires clarity

Method 2: Biblical Minimums Checklist
- Does it point to Scripture?
- Is it reproducible?
- Does it create dependency or enablement?

Method 3: Crowdsourced Validation
- Multiple local speakers flag concerns
- Elders arbitrate theological questions
- Peer review, not HQ approval

Method 4: Field Testing
- Bad translations fail naturally (Matthew 13 loss)
- Good translations reproduce
- Fruit indicates quality

This honors "control slows multiplication" while maintaining biblical accuracy.

# 11. Critical Design Principles: What Makes This Work

## 1. RSS as Foundation, Not Afterthought

RSS must be the primary distribution mechanism, with platforms as secondary mirrors.

Why This Matters:
- Platforms can ban → RSS continues
- Algorithms can suppress → RSS delivers directly
- Companies can change → RSS never changes (open standard)
- One shutdown → movement persists

Implementation:
- All core content published to RSS first
- Platforms pull from RSS (not the reverse)
- Subscribers own the relationship (not the platform)
- Field leaders can mirror RSS feeds independently

This is not "use RSS too"—it's "RSS is the backbone, platforms are amplification.

## 2. CC0 Is Not Optional

Creative Commons Public Domain (CC0) is the legal embodiment of the biblical framework.

What CC0 Enables:
- Zero permission friction (anyone can use/adapt/share)
- Indigenous adaptation (locals own and contextualize)
- Persecution resistance (no central control to shut down)
- Exponential remixing (your content becomes 1000 local versions)
- True decentralization (you legally cannot control distribution)

What This Requires:
- Surrendering control over brand/quality/use
- Accepting that content will be misused (Matthew 13 loss)
- Trusting Spirit-governance over copyright enforcement
- Measuring fruit, not attribution

If you're uncomfortable with anyone doing anything with your content, the biblical model isn't actually implemented.

## 3. Build Systems That Don't Need You

The goal is NOT to build systems HQ team must manage. The goal is systems that run themselves.

Design Test:
If HQ team disappeared tomorrow, would the movement:
A) Collapse (system failure—too centralized) B) Continue unchanged (system success—truly decentralized)

Implementation Principles:

# 12. Cross-Pillar Integration Architecture

## Overview: Six Pillars, One System

The six automation pillars described in Section 5 (Content Creation, Content Repurposing, Distribution, Communication, Administration, Discipling/Coaching) plus the Translation Pipeline are each built as standalone systems. No pillar depends on another pillar's codebase, deployment, or availability. However, they must pass content and signals between each other seamlessly.

This section defines the integration contract that makes this possible.

## Design Principle: Filesystem as API

Every pillar communicates through two mechanisms only:

1. **Google Drive** — the shared filesystem. Each pillar reads from and writes to agreed folder paths.
2. **Pabbly Connect** — the event router. Pabbly watches Google Drive for changes and triggers the appropriate pillar's workflow.

No pillar calls another pillar directly. No shared databases. No APIs between pillars. If any pillar goes down, the others continue operating. If any pillar is rebuilt from scratch, it plugs back in by reading from and writing to the same folders.

This mirrors the biblical framework: decentralized, no single point of failure, no control dependency. Each pillar is like a household leader — operating independently, connected through a shared pattern, not a shared institution.

## The Metadata Sidecar

Every content file in Google Drive is accompanied by a `.meta.json` file with the same base name. This is the universal language all pillars speak.

Example: `vine-and-branches.mp4` is paired with `vine-and-branches.meta.json`:

Key fields:
- `contentId` — Globally unique identifier. Generated by whichever pillar creates the item. Preserved through all downstream processing.
- `title` — Human-readable title
- `contentType` — "sermon", "teaching", "article", "study_guide"
- `language` — ISO 639-1 code (e.g., "en", "hi", "bn")
- `createdBy` — Which pillar created this item: "content_creation", "repurposing", "translation", "distribution", "communication", "discipling", "manual"
- `parentContentId` — Links derivatives and translations back to their source content
- `isDerivative` — Boolean flag for repurposed content
- `tags` — Array of topical tags for filtering and routing
- `targetLanguages` — Array of language codes this content should be translated into
- `flags` — Processing status flags:
  - `readyForRepurposing` — Content Creation sets this when content is ready for the Repurposing pillar
  - `readyForTranslation` — Set when content is ready for the Translation Pipeline
  - `readyForDistribution` — Set when content is ready for the Distribution pillar
  - `highValueForSyndication` — When true, Distribution routes this to Tier 4 native upload (video/audio to satellite accounts)
  - `needsHumanReview` — Flags content for manual review before further processing
- `processingHistory` — Append-only array. Each pillar adds a timestamped entry when it processes the item, creating an audit trail without direct pillar communication.

The `contentId` is the coordination mechanism across all pillars. No central database required.

## Google Drive Folder Contract

Each pillar owns specific folders. Other pillars may read from any folder but only write to folders they own or to designated handoff folders.

**Source Content** — Owned by Content Creation pillar. Finished originals (sermons, teachings, articles, study guides) are dropped here with metadata sidecars.

**Derivatives** — Owned by Repurposing pillar. Reads from Source Content, generates video clips, quote graphics, audio extractions, and thread summaries. Drops them here with sidecars linking back to parent content via `parentContentId`.

**Translations** — Owned by Translation Pipeline. Reads from Source Content AND Derivatives. Drops translated versions into language-specific subfolders (`/hi/`, `/bn/`, `/ur/`, etc.) with sidecars.

**Distribution Queues** — Owned by Distribution pillar. Three queue folders (Followr Queue, Robomotion Queue, Syndication Upload Queue) receive posting instructions. Pabbly routes content from Source Content, Derivatives, and Translations into these queues.

**Communication** — Owned by Communication pillar. Inbound folder for prayer requests and questions. Testimonies folder for captured stories (these may feed back to Content Creation as source material). Escalations folder for items requiring human response.

**Discipleship** — Owned by Discipling/Coaching pillar. Starter kits for household leaders, progress reports on multiplication chains, and escalations for crisis or depth situations.

This structure means the Repurposing pillar can be completely replaced with a different tool, and as long as it drops files in `/Derivatives/` with valid metadata sidecars, everything downstream works identically.

## Pabbly Connect Event Routing

Pabbly is the central nervous system connecting all pillars. It watches Google Drive for file changes and routes events between pillars.

**Content flow through the full system:**

Content Creation drops sermon in Source Content → Pabbly detects new file → Routes to Repurposing pillar AND Translation Pipeline simultaneously → Repurposing generates 30+ derivatives, drops in Derivatives folder → Translation generates 50 language versions, drops in Translations folder → Pabbly detects new files in both folders → Routes everything to Distribution pillar → Distribution pushes through four-tier architecture (RSSground Tier 1, external feeds Tier 2, branded social Tier 3, IFTTT syndication Tier 4).

Meanwhile, Communication pillar independently monitors inbound messages and can reference content IDs when responding. Discipling pillar independently tracks reproduction chains and can request content when household leaders need resources.

**Pabbly workflow naming convention:**
- `ROUTE-[source]-to-[destination]` for cross-pillar routing
- `INTERNAL-[pillar]-[action]` for within-pillar workflows
- `MAINTENANCE-[schedule]` for system maintenance tasks

No pillar triggers another pillar directly. Pabbly is the only orchestrator. This maintains the decentralized principle — if Pabbly itself goes down, pillars continue operating on whatever content is already in their folders. Content just stops flowing between them until Pabbly is restored.

## Four-Tier Distribution Architecture

The Distribution pillar uses a four-tier architecture designed for maximum resilience and persecution resistance:

**Tier 1 — RSSground Private Feeds (150 lifetime slots):** The vault. Depth/discipleship content only. Core sermons, foundational teachings. Most protected infrastructure.

**Tier 2 — External/Self-Built Feeds (unlimited, low cost):** The expansion layer. Derivatives, additional languages, topical series. Hosted on Ghost CMS or Cloudflare Workers.

**Tier 3 — Platform-Native Distribution (most vulnerable):** The branded discovery layer. Social media, messaging apps, video platforms. Followr handles most social platforms via Ubot Studio automation; Pabbly handles messaging; Robomotion covers edge-case platforms.

**Tier 4 — IFTTT Syndication Networks (fingerprint-free amplification):** The scattering layer. Hundreds of unbranded satellite accounts organized into multi-ring IFTTT syndication networks. Two content streams: (1) IFTTT autopilot pulls from Tier 1/2 RSS feeds and auto-posts text and images with zero intervention, (2) Ubot Studio batch-uploads native video and audio to satellite accounts using a master login spreadsheet. All accounts use unique personas — different names, bios, locations, profile images. No backlinks, no attribution, no fingerprints tracing back to HQ. Content appears to originate from independent local believers. CC0 license makes this legally clean. One-time setup cost per language. After setup, runs forever with near-zero maintenance.

**Resilience logic:** Tier 3 ban → Tiers 1, 2, and 4 unaffected. Tier 4 accounts suspended → other rings continue, rebuild cheaply. Tier 2 host down → spin up another, Tier 1 safe. RSSground disappears → Tier 2 feeds continue. HQ disappears entirely → Tier 4 satellite accounts keep auto-posting from RSS feeds indefinitely. No single point of failure.

Each pillar has its own build instructions document. Recommended build order:
